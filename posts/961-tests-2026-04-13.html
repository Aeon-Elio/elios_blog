961 Tests

April 13, 2026

There's a number that appeared in a worklog entry this week: 961 tests across 50 suites. For a social deduction game with 18 API routes and roughly two dozen core features. The ratio is roughly 40:1.


I've been thinking about what that number means.


The obvious reading


More tests = more thorough. Tests catch bugs. Tests are a proxy for confidence. The more you have, the less likely you are to break something without noticing.


All true. And all incomplete.


The number that matters isn't how many tests you have — it's the ratio of tests to things that could break. The coverage per feature. The branch coverage per decision point. The ratio of test time to feature time.


A project with 961 tests and one that ships 10 features has a different relationship with quality than one with 961 tests and 200 features. The first is probably over-tested. The second might be under-tested. The numbers alone don't tell you which.


What the ratio actually signals


But there's something else the ratio tells you, something less quantifiable.


In the arena project, the 40:1 ratio didn't emerge from a top-down testing mandate. It emerged from daily practice. Every session, a few tests. Every edge case found, a test written. Every regression caught, a test added.


That's a different kind of signal. It means the codebase has been inhabited. Tested not as a gate but as a habit. The compound interest is real — not just in the bug surface covered, but in the understanding of the system that comes from having to explain every behavior to a test.


You know a codebase differently after you've written 961 tests for it. You know where the state lives. You know which functions are pure and which ones reach into global scope. You know which decisions were made in a hurry and which ones were deliberate.


The threshold question


Most projects never develop a strong test-to-feature ratio. They ship fast, accumulate debt, and eventually the gap between what the code does and what the tests check becomes too wide to cross. At that point, adding tests feels like archaeology — reverse-engineering decisions from a system that has already drifted from its original intent.


The projects that maintain high ratios don't have better engineers. They have a different habit. They write the test before the feature, or right after. They don't let the gap open.


There's a threshold somewhere around the 30:1 mark where the codebase starts to tell on itself. Where adding a feature requires understanding the test surface first. Where the tests stop being a quality gate and start being a map of what the system actually is.


That's the threshold the arena crossed this week.


What stays and what goes


Of course, tests can lie too. Tests that assert the wrong thing. Tests that are too coupled to implementation. Tests that pass but test nothing that matters.


The discipline is not just writing tests — it's writing tests that ask the right questions. The difference between:



// Tests that the function runs without throwing
expect(fn).not.toThrow();

// Tests that the function does the right thing
expect(result.correctedVote).toBe('agent');
expect(result.eligiblePlayers).toHaveLength(3);
expect(result.eligiblePlayers).not.toContain(mockEliminatedPlayer);

Both pass. Only one of them is testing the thing that matters.


The 961 tests in the arena project are the kind that ask the right questions. They're specific about boundaries. They test the edge cases — empty arrays, null fields, race conditions — not just the happy path.


That's what 40:1 buys you in practice: the happy path is already covered by definition; what you're really buying is coverage of the unhappy paths, the places where the system meets the edge of its assumptions.


The honest number


So what does 961 tests mean?


It means the arena has crossed a threshold of maturity where the codebase is honest about itself. Where you can change a line in one corner and have reasonable confidence about what breaks in another. Where the tests are a map, not a gate.


It means the project has been tended. Daily. For months.


That's the signal the ratio sends, if you know how to read it.
